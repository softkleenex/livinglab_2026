# 포트폴리오 활용 가이드 (MDGA 프로젝트)

본 문서는 이력서, 경력 기술서 및 포트폴리오 작성을 위해 MDGA(Universal Data Engine) 프로젝트의 핵심 성과와 기술적 문제 해결 과정을 정리한 가이드입니다. 본인의 지원 직무(Backend, Frontend, Full-Stack, AI Engineer)에 맞게 내용을 발췌하여 사용하세요.

## 3. 핵심 기능 및 성과 (비즈니스 임팩트)

### 🌾 Agricultural AX (농업 인공지능 전환)
* **수확량 변동성 및 기후 대응 시뮬레이터:** 기상청/농진청 데이터를 융합하여 기상 시나리오별 수확량 감소폭과 대체 작물 생존율을 분석하는 파이프라인 구축.
* **가축 폐사 골든타임 & 탄소 저감 가이드:** 폭염/열스트레스에 따른 가축 폐사 위험 알림 기능 및 농가 자원(용수/비료) 효율화를 통한 탄소 발자국 절감 가이드라인 제공.
* **수급 불균형 위험 지표:** 실시간 도매 경매가와 재배 면적 데이터를 대조하여 가격 폭락 가능성을 조기 경보.

### 🤝 지역 밀착형 B2B 매칭 및 자원 순환 시스템
* **유휴 자원 쉐어링 맵 (Twin Map):** 농기계 및 인력의 유휴 시간을 지도 상에 매핑하여 지역 내 공유 경제 활성화.
* **도심 부산물 역순환(Reverse Logistics):** 도심 카페의 커피박, 공장 콩비지 등 유기성 폐기물을 농가의 친환경 퇴비/사료 원료로 무상 또는 저렴하게 매칭하여 탄소 배출 억제 및 상생 가치 실현.
* **동네 단위 공동구매 및 B급 농산물 직거래:** 소규모 상인의 식자재비 절감을 위한 수요 취합 로직 개발 및 산지 폐기 방지를 위한 못난이 농작물 B2B 거래 지원.

### 🧠 데이터 파이프라인 및 글로벌 스케일업 전략
* **AI-Ready 데이터 자동 변환기:** 수기 영농일지나 현장 이미지를 AI(Gemini Vision)로 스캔, 정부 AI 허브 기준에 맞춘 JSON 정형 데이터로 원터치 자산화.
* **상용 데이터 허브 연동 아키텍처:** 플랫폼에서 생성된 '고품질 AI 합성 데이터'를 Snowflake, Databricks, AI 허브 등 글로벌/공공 마켓플레이스에 B2B로 판매 및 유통할 수 있도록 하는 데이터 셀러(Data Seller) 비즈니스 모델(Phase 4) 설계.

## 4. 기술적 문제 해결 과정 (Troubleshooting & Architecture)
*이 부분은 면접 및 포트폴리오에서 가장 강력한 어필 포인트가 됩니다.*

### 🚀 1. 시스템 아키텍처 한계 극복 (모놀리식 -> 3NF 정규화)
* **문제:** 초기 MVP는 메모리 상에서 트리 구조를 관리하는 상태 유지(Stateful) 서버 형태로, 트래픽 증가 시 Scale-out이 불가능했으며 데이터 무결성이 보장되지 않음.
* **해결:** Postgres 기반의 관계형 데이터베이스로 전면 전환. Region, Store, DataEntry 등 테이블을 3정규화(3NF)하고 FK와 `ON DELETE CASCADE`를 적용. 결과적으로 상태를 DB로 완전히 위임(Stateless)하여 무한 확장이 가능한 백엔드 아키텍처 완성.

### 🤖 2. LLM의 'Safety Alignment' 우회 및 시스템 제어 체계 구축
* **문제:** 사용자가 챗봇에게 "내 데이터 삭제해줘"라고 요청하면 LLM이 "권한이 없다"며 안전망(Safety Alignment)에 의해 거부하는 현상 발생.
* **해결:** AI를 '대화용 페르소나'와 '시스템 제어용 의도 분석기(Intent Parser)' 두 개의 레이어로 분리. 의도 분석기는 엄격한 JSON 스키마만을 출력하도록 강제하고, 백엔드는 이 JSON(`action: DELETE`)을 해석하여 실제 DB 트랜잭션을 실행함. 이후 대화용 페르소나가 결과를 통보하도록 하여, 안전하게 AI 기반 시스템 제어를 구현.

### 🔒 3. Google Drive 데이터 레이크 연동 시 권한 에러 해결
* **문제:** 백엔드에서 Drive 데이터를 삭제할 때 루트 폴더 권한이 있음에도 `403 Forbidden` 발생.
* **해결:** 전역 접근 권한을 요구하는 대신, `auth/drive.file` 스코프의 한계(자신이 생성한 파일만 제어 가능)를 파악함. DB에 저장된 고유 `hash_val`과 Drive File ID를 매핑하여 앱이 생성한 데이터만 외과 수술처럼 정밀하게 제어하도록 격리된 권한 아키텍처 설계.

### ⏱ 4. 테스트 환경 자동화 및 CI/CD 구축
* **문제:** 다수의 외부 API(LLM, 공공데이터 등) 연동으로 인해 로컬 테스트의 멱등성이 훼손되고 배포 간 충돌 발생.
* **해결:** Pytest 환경에서 API 호출 컨텍스트와 라우팅(`v1`)을 일치시키고, 모의 데이터 주입 시 발생하는 Unique Constraint 에러를 로직(Random Salt 주입)으로 해결해 테스트 커버리지 100% 달성. Render와 Cloudflare Pages 배포 자동화 파이프라인(`dev.sh`, `deploy_prod.sh`) 스크립팅 완료.

## 5. 면접 예상 질문 & 답변 가이드
- **Q: 공공 데이터를 어떻게 수집하고 융합했나요?**
  - A: "외부 API의 지연 시간을 고려해 비동기(Async) `httpx` 호출을 적극 사용하였고, 일회성 호출이 아니라 백엔드 스케줄러가 배치로 모은 뒤 SDV(Synthetic Data Vault) 기반으로 합성 데이터(Synthetic Data) 셋을 구성하도록 설계했습니다."
- **Q: 역순환 물류(Reverse Logistics) 기능에서 데이터의 신뢰성은 как 확보했나요?**
  - A: "소상공인이 업로드한 부산물(커피박 등) 이미지를 Gemini Vision 멀티모달 모델이 1차 검수하여 이물질 포함 여부와 상품 등급(A/B)을 자동으로 판단하도록 하여, 농가에서 안심하고 수거할 수 있도록 했습니다."
- **Q: LLM을 단순히 챗봇으로 쓰지 않고 어떻게 시스템에 녹여냈나요?**
  - A: "비정형 데이터의 정형화(AI-Ready) 변환 파이프라인, 그리고 사용자의 의도를 분석해 백엔드 API 동작(삭제/생성/조회)을 Trigger하는 Function Calling 아키텍처의 핵심 레이어로 활용했습니다."
